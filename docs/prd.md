## Problem Statement

中小团队（10-20人）的排班管理完全依赖人工：管理者需要收集每位员工的下月时间约束（哪天有事、偏好什么班次），再按排班规则手工排表。这个过程耗时、容易出错（如遗漏员工的请假请求、违反连续工作天数限制），且员工无法随时查看最新排班。

员工端缺乏一个简单、低门槛的方式提交需求——他们应能像聊天一样用自然语言说出约束，而不需要填写复杂表单。管理者需要一键生成排班、灵活微调、最终发布的能力。

## Solution

一个 **Web 应用**（Java Spring Boot + React），分员工端和管理员端：

- **员工端**：用自然语言在对话框中输入下月排班需求（如"2月3号有事、偏好早班"），由 DeepSeek 大模型解析为结构化数据；月历视图查看已发布排班。
- **管理员端**：汇总所有员工需求，一键触发 LLM 根据硬性规则 + 员工偏好生成完整排班表；表格视图中手动调整后发布；发布后员工立即可见。

核心差异化：用 LLM 替代传统排班算法，让规则配置和需求理解都更灵活。

## User Stories

### 员工端

1. 作为员工，我希望能用自然语言（如"下个月3号有事"）在对话框中描述我的排班需求，而不是填复杂的表单
2. 作为员工，我希望提交需求后，系统能展示它理解的结果（如"已明白：2月3日不可上班，偏好早班"），让我确认或修改
3. 作为员工，我希望可以查看和编辑我已提交的当月需求
4. 作为员工，我希望能看到已发布的下月排班表，以月历形式展示每天是谁上什么班次
5. 作为员工，我希望排班表用颜色区分班次（早班绿/晚班蓝/休息灰），一眼能看清
6. 作为员工，如果排班尚未发布，我希望能看到"尚未发布"的明确提示
7. 作为员工，我不想看到其他人提交的排班需求（隐私）

### 管理员端

8. 作为管理员，我希望能在一个列表里看到所有员工本月提交的需求摘要（谁、哪天不能来、偏好什么班次）
9. 作为管理员，我希望能一键触发 AI 生成下月排班，系统自动考虑规则 + 所有人偏好
10. 作为管理员，我希望生成中能看到进度提示，知道系统正在计算
11. 作为管理员，我希望能看到 AI 生成后的排班概览统计（每人上班天数、早/晚班分布）
12. 作为管理员，我希望在发布前能在表格中手动调整任何员工的任何一天班次（早班/晚班/休息下拉切换）
13. 作为管理员，我希望手动调整后的排班能保存为草稿，随时回来继续编辑
14. 作为管理员，我希望点击"发布"后，所有员工端立即可见最新排班，且表格锁定
15. 作为管理员，我希望发布后仍可"取消发布"回去修改
16. 作为管理员，我需要系统强制执行硬性排班规则（见下方），不应产生违规排班

### 系统规则

17. 作为系统，我必须保证同一员工不会在同一天被安排两个班次
18. 作为系统，我必须保证同一员工不会连续上班 5 天或更多
19. 作为系统，我必须保证每天每个班次（早班、晚班）至少有 1 人
20. 作为系统，我必须保证法定节假日所有人都不排班
21. 作为系统，我必须保证每人每月上班天数不超过 20 天
22. 作为系统，我应在满足硬性规则的前提下，尽量满足员工的日期偏好和班次偏好

### 通用

23. 作为用户，我希望能用账号密码登录，登录后根据角色自动进入对应界面
24. 作为管理员，我希望能手动添加员工账号
25. 作为用户，如果 AI 服务暂时不可用，我希望看到友好提示而不是报错崩溃

## Implementation Decisions

### 技术架构
- **架构模式**：Spring Boot 3 一体式应用，React SPA 打包为静态资源内嵌在 jar 中，单文件部署
- **数据库**：H2 内嵌文件数据库 — 零安装，10-20 人规模足够
- **前端**：React (Vite) + FullCalendar 日历组件
- **认证**：Spring Security + Session，登录后按角色路由

### 数据模型（3 张表）
- **user**：id, username, password(BCrypt), name, role(EMPLOYEE/ADMIN)
- **shift_requirement**：每个员工每月一条，存储原始输入 + LLM 解析后的结构化需求
- **schedule**：每个员工每天一条排班记录，含状态(DRAFT/PUBLISHED)

### LLM 集成
- **两次 DeepSeek API 调用**：① 解析员工自然语言 → JSON 结构；② 规则 + 需求 → 完整排班表
- **容错**：解析失败提示重试；生成不满足约束自动重试 1 次
- **Prompt 设计**：硬约束作为"必须遵守"规则，员工偏好作为"尽量满足"软约束

### API 设计
- RESTful 风格，/api/** 为后端接口，其他路径交由 React SPA 处理
- 管理员端 API 统一在 /api/admin/** 下，需 ADMIN 角色
- 排班生成和发布操作需幂等保护

### 业务规则
- **硬约束（系统强制校验）**：不连上两班、不连上 5 天、每班至少 1 人、节假日全休、月 ≤20 天
- **软约束（LLM 尽量满足）**：员工不可上班日期、班次偏好、上班天数尽量平均

## Testing Decisions

- **测试原则**：测试外部行为，不测试实现细节
- **核心验收**：一条命令启动 → 浏览器可访问 → 主链路走通（员工输入需求 → 管理员生成排班 → 调整 → 发布 → 员工查看）
- **规则验证**：3 员工 × 7 天小数据集，人工验证 5 条硬约束
- **边界测试**：全员请假、零需求、API 不可用等场景
- **不做**：单元测试、集成测试自动化（一天时间限制）

## Out of Scope

- 手机 App / 小程序
- 邮件/消息通知（排班发布后通知员工）
- OAuth / SSO 第三方登录
- 排班历史记录对比和版本回滚
- 多语言支持
- 员工自主换班 / 申请调班
- 考勤打卡
- 工时统计报表导出
- Docker / K8s 容器化部署
- CI/CD 流水线

## Further Notes

- 排班生成依赖 DeepSeek API，需确保 API Key 有效且额度充足
- 系统启动时自动创建默认管理员账号（admin/admin123）和预置中国法定节假日
- 员工账号由管理员手动创建
- 前端开发需 Node.js，但运行时只需 Java 17+
- 项目只有一天开发时间，需严格控制范围，优先保证核心闭环可用
